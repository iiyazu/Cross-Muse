# xmuse v0.4.0

`v0.4.0` is a production-evidence release for the loopback-only, single-user Room Workroom.
It preserves `chat.db` as the authority for Room, execution, and memory-governance facts; the
MemoryOS index remains derived and rebuildable.

## Trusted optional MemoryOS companion

Memory remains optional at installation. The companion is built from `memoryos-lite 0.2.1`
at clean source SHA `26a77ece3bfe865169890a7dd49b5076c13ab723`. When the installer-owned
manifest, relative executable, and payload digests verify, Workroom's default
`--memory-mode auto` starts the full-local companion. Missing companions leave Room Runtime
available with memory marked not installed; invalid companions fail closed and are not run.
Workroom never searches PATH or arbitrary configuration for a sidecar. `--memory` remains the
explicit `on` compatibility alias and `--no-memory` selects `off`.

Full-local keeps offline BM25, FastEmbed, RRF hybrid retrieval, paging, and
external-governance advisory generation. Recall evidence remains bounded and re-proved against
`chat.db`; MemoryOS cannot approve or write cross-Room authority directly.

The release qualification settled 48 correlations and recorded six non-empty recall receipts,
19 re-proved source references, and 16 derived `page`/`recall` items. It covered approved
project-rule and user-preference sharing, non-recent Room decisions, lexical and semantic
queries, causal/current-correlation exclusion, unapproved-candidate exclusion, and MemoryOS
kill/replay/capability recovery.

Safe `room_memory_diversity_result/v1` SHA-256:
`79356275b4ca9fbec3e856531d6cd8fc45404782ddc512d88c2c14f5192bc183`.

The result contract contains aggregate counts, opaque references, reason codes, and digests
only. It excludes Room text, provider output, MemoryOS internal IDs, paths, keys, and traces.

## Closed Harness profile matrix

The Harness now supports the existing `docs/v1`, `python-uv/v1`, and `xmuse-monorepo/v2`
profiles plus `python-uv-ty/v1`, `node-pnpm-library/v1`, and
`node-pnpm-next-workspace/v1`. Each profile owns its markers, lock/configuration checks,
direct read-only entrypoints, and capability proof. xmuse does not run repository scripts,
accept arbitrary gate argv, or install dependencies over the network to make a repository fit.

The release matrix promoted fixed patches in memU, MemoryOS, and twg. Clowder, Letta, and
mem0 correctly failed closed when their fixed profile capability was not satisfied.

Safe matrix SHA-256:
`21a4a5fd12ea3a9eef21d76ef6cea95981f5c889bb5a00bb4acf450ae4ca509c`.

## Execution boundary and Workbench

Execution capability consumers now receive direct operator, controller, runtime, read, or
review Store access rather than an internal broad Store facade. This preserves caller-owned
transactions, authorization, promotion, recovery, backups, and public projections.

The Workbench adds a readiness-first onboarding path and progressive disclosure. The Room
timeline remains the shared fact stream; the Agent, Room, and Runtime dock shows only safe,
bounded native Codex activity and backend-proven actions. It does not recreate Codex state,
show raw thinking or tool payloads, or add a Dashboard.

## 45-minute release-candidate soak

The release-candidate qualification exercised three Rooms, four persistent Agents, and six
waves for more than 45 minutes. It settled 18/18 correlations in 159 attempts while injecting
provider, Room Runner, MemoryOS, and stream-cache faults. There were no duplicate outcomes,
cross-Room leaks, orphaned providers, recovery residue, workspace changes, or browser console
errors.

Safe soak SHA-256:
`3e37af5a4866663183ab6cc2c0625b38ad7fb57f87dadcf99904d373c7031553`.

Raw dialogue, databases, logs, screenshots, provider output, and runtime paths are not release
artifacts.

## Supported boundary

- Linux x86_64 / WSL with CPython 3.11;
- loopback-only, single-user deployment with an authenticated local Codex CLI;
- optional, trusted full-local MemoryOS companion;
- Room Codex remains read-only, network-disabled, and limited to the durable outcome MCP tool;
- Google A2A remains retired; xmuse does not provide remote authentication, a central speaker
  queue, Dashboard, general terminal, or general process manager.
