# xmuse 解耦开发协议

更新日期: 2026-06-02

## 目标

本协议用于把 xmuse 后续开发拆成可并行推进的层面，同时避免形成第二套状态源，或
让前端、dashboard、workflow、GOD runtime 互相硬耦合。

当前协议承认两种状态:

- **目标态**: graph-set / lane graph / durable event 是业务权威，runner 只消费
  ready work。
- **迁移态**: `feature_lanes.json` 仍是当前 runner、state machine、dashboard、
  MCP 工具共同读写的 live queue/status source。它不是最终设计权威，但在 cutover
  前必须被当成当前执行事实源保护。

## 快速浏览

- 讨论层去中心化：GOD/peer 对等质询，目标是产出可靠 blueprint。
- 执行层中心化：一个 xmuse coordinator 负责 2a/2b/2c 调用、subagent 编排和状态写入。
- subagent 是工具：由 coordinator 传入上下文和验收标准，输出证据，不直接写状态。
- Ray 是 runtime backend：可以承载 coordinator/GOD/worker 生命周期，不拥有状态。
- LangGraph 是 workflow backend：可以编排 2a/2b/2c，不直接写 lane status。
- `feature_lanes.json` 当前仍是执行事实源；cutover 后才退化为兼容导出。

## 术语表

| 术语 | 定义 |
|---|---|
| mission | 用户提出的顶层目标或主题 |
| blueprint | GOD 群聊把 mission 固化后的目标、边界、验收标准和非目标 |
| feature plan | 从 blueprint 拆出的中等粒度 feature 列表及 feature 间依赖 |
| lane graph | 单个 feature 内部的小任务 DAG |
| graph-set | 一个 feature plan 对应的一组 lane graph，跨 feature 可并行 |
| lane | runner 可执行的最小任务切片 |
| decentralized peer discussion | 讨论阶段多个 GOD/peer 对等发言、互相质询、共同收束蓝图 |
| xmuse coordinator | 统一驱动 2a/2b/2c 的逻辑控制者，负责事件消费、上下文装配、subagent 调用、证据收集和状态写入 |
| centralized coordinator | 执行阶段采用单一 xmuse coordinator 仲裁状态和调度，避免多个自治 agent 彼此协商 |
| CLI subagent harness | Codex/OpenCode 等 CLI 已有的子任务执行机制，由 coordinator 受控调用 |
| skill | 可复用的执行约束、检查清单或 runbook，用来约束 subagent 行为 |
| event | 跨层状态变化通知，至少一次投递，消费者必须幂等 |
| artifact | 可持久化追溯的业务对象，例如 blueprint、feature plan、graph-set |
| artifact ref | 指向 artifact 的稳定引用，不等同于具体文件路径 |
| read model | 从权威状态派生的只读视图，给 TUI/dashboard 消费 |
| read envelope | 面向 TUI/API 的聚合响应，通常包含 worklist、cards、run health |
| card | 群聊流中的 compact UI 单元，可 drill-down 到 dashboard |
| projection | 从权威 artifact 派生给 legacy runner 消费的兼容视图 |

## Agent 拓扑原则

xmuse 不把所有阶段都设计成同一种多 agent 拓扑。不同阶段的目标不同，拓扑也不同:

| 阶段 | 推荐拓扑 | 目的 |
|---|---|---|
| 需求讨论 / blueprint | 去中心化 GOD/peer 群聊 | 通过多视角质询获得更客观可靠的蓝图 |
| feature/lane 拆解 | 中心化 workflow + 可调用 reviewer | 保证依赖、粒度、artifact refs 和验收标准稳定 |
| 代码执行 / 测试 / 修复 | 中心化 coordinator + CLI subagent harness | 提高上下文一致性、减少 merge/rework 摩擦 |
| review / takeover | 中心质量门禁 + 可调用审查/修复 subagent | 保证验收对齐和状态写入一致 |

原则:

- 去中心化只作为讨论和蓝图审议阶段的默认形态。
- 执行开发阶段默认中心化，不把每个 worker/reviewer 都提升为相互独立的自治 GOD。
- subagent 是 coordinator 的受控工具；它不拥有 lane 状态，也不直接写 durable store。
- 可复用执行规范优先沉淀为 skills，而不是新增常驻 agent 角色。
- Ray 可以管理 coordinator / GOD / worker runtime 生命周期，但不改变状态权威。
- LangGraph 可以编排 workflow，但不改变“执行阶段由 coordinator 控制”的原则。

## Coordinator 形态与生命周期

`xmuse coordinator` 是逻辑角色，不是新的去中心化 GOD。它统一驱动 2a/2b/2c:

```text
blueprint.approved
-> coordinator 调用 Layer 2a
-> feature_plan.ready
-> coordinator 调用 Layer 2b
-> graph_set.ready / lane.ready
-> coordinator 调用 Layer 2c
-> worker/review/rework/takeover
-> run.terminal / blueprint.gap_found
```

Stage 0 当前形态:

- coordinator 是单 native Python 运行进程，主要由 `xmuse/platform_runner.py` 承载。
- `platform_runner.py` 管理 writer lease、tick loop、blueprint automation、
  chat driver、peer scheduler 和 lane dispatch。
- `PlatformOrchestrator` 是 lane execution / review / merge / status transition 的
  主要执行控制面。
- `BlueprintAutomationService` 是 2a 入口的一部分；feature planning 和 graph-set
  生成应作为 coordinator 调用的 service，而不是独立自治 agent 链。

Stage 1/2 可替换形态:

- coordinator 可以包装成 CLI 命令，适合本地可调试运行。
- coordinator 可以包装成 Ray Actor，适合长驻生命周期、崩溃重启和并发 actor 管理。
- coordinator 可以用 LangGraph backend 编排 2a/2b/2c 的节点顺序。
- 无论形态如何，只有 coordinator 或它调用的 state-machine service 能写 execution
  状态；subagent、Ray actor 子 worker、LangGraph node 都不能直接写 lane status。

生命周期规则:

- 同一个 workspace/run scope 同时只能有一个 active coordinator writer。
- coordinator 启动时必须恢复 durable events、planning runs、graph-set refs 和
  current projection revision。
- coordinator 停止时不得丢失已领取事件；未 ack 事件按 at-least-once 规则重放。
- coordinator 崩溃恢复后通过 idempotency key 和 projection revision 去重。

## 分层边界

### 1. GOD 群聊层 + TUI

职责:

- 创建和切换 conversation/workspace。
- 展示人类与多个 GOD/peer 的自然语言讨论；讨论阶段允许 peer-to-peer 质询和
  @mention。
- 展示 compact cards: blueprint、feature plan、lane graph、run progress、review
  verdict、takeover。
- 通过点击 card 跳转 dashboard drill-down。

可依赖:

- `chat.db` 中的 conversation/message/participant/inbox。
- execution card/read envelope API。
- 只读 run health、feature graph set、lane graph summary。

禁止:

- 直接写 `feature_lanes.json`。
- 直接修改 runner 状态。
- 把 dashboard API 当成主流程驱动器。
- 把讨论层 peer-to-peer 拓扑延伸为执行层的默认拓扑。

测试边界:

- 单层测试使用 fake chat store、fake read envelope 和 fixture cards。
- 不要求真实 runner、Ray actor 或 LangGraph。
- TUI contract test 固定读取 `conversation + messages + cards + worklist envelope`
  fixture，验证未知 card 可降级渲染。

### 2a. Blueprint Decomposition

职责:

- 由 xmuse coordinator 消费 `blueprint.approved` 并调用 2a service。
- 创建或恢复 planning run。
- 产出 feature plan proposal。
- 调用 planner/reviewer GOD、fake planner 或中心化 planning coordinator。
- 发出 `feature_plan.ready`、`planning.failed` 或 `blueprint.rejected`。

可依赖:

- planning event store。
- planning run store。
- feature plan store。
- planner/reviewer adapter。

禁止:

- 直接投影 lane。
- 直接启动 runner。
- 直接修改 lane status。

测试边界:

- 使用 fake blueprint loader、fake planner、fake reviewer。
- Contract test 验证同一 `blueprint.approved` 重放不会创建重复 planning run。
- 不依赖 lane graph generator 或 runner。

### 2b. Lane Graph Generation

职责:

- 由 xmuse coordinator 消费 `feature_plan.ready` 并调用 2b service。
- 为每个 feature 生成 lane graph DAG。
- 生成 graph-set artifact。
- 验证 feature 依赖和 lane 依赖是 DAG。
- 发出 `graph_set.ready` 或 `graph_set.failed`。

可依赖:

- feature plan store。
- lane graph / graph-set artifact store。
- decomposition review。

禁止:

- 绕过 graph-set artifact 直接手写一批 flat lanes 当权威图。
- 直接调度 worker。
- 直接合并代码。

测试边界:

- 使用 fixture feature plan 和 fake graph-store root。
- Contract test 验证 graph-set id、feature dependency、lane dependency、artifact refs
  稳定。
- 不依赖 GOD 群聊、TUI 或真实 runner。

### 2c. Lane Execution Scheduling

职责:

- 由 xmuse coordinator 消费 `graph_set.ready` 或 `lane.ready`。
- 将 ready lanes 投影到当前 runner 可消费的队列。
- 由中心化 execution coordinator 调度 runner/orchestrator 执行 lane。
- coordinator 可以调用 CLI subagent harness 执行具体代码、测试、修复。
- 汇总 worker、review、rework、takeover 结果。
- 发出 `lane.updated`、`lane.blocked`、`run.terminal`。

可依赖:

- graph-set store。
- 当前迁移期的 `feature_lanes.json`。
- `LaneProjectionSyncer`、`LaneStateMachine`、`PlatformOrchestrator`。
- provider policy 和 worker backend。
- CLI subagent harness。
- execution/review/takeover skills。

禁止:

- 让 Ray actor 或 LangGraph 直接写 lane status。
- 让 ephemeral subagent 直接写 lane status 或 durable store。
- 把多个相互独立 worker 串成必须彼此协商的执行链。
- 绕过 state machine 改写 execution 状态。
- 多 runner 抢写同一个 `feature_lanes.json`。

测试边界:

- 使用临时 lanes file、fake worker transport、fake subagent harness、fake review verdict。
- Contract test 验证投影幂等、依赖 ready-set、state transition guard。
- Runner 集成测试只覆盖 compatibility projection，不要求真实 GOD 群聊。

### 3. 层间信息交流和对接契约

职责:

- 定义跨层事件和 schema。
- 定义 read model、card、artifact ref 的稳定字段。
- 定义 test double、contract test 和兼容性验证方式。
- 保证 TUI、workflow、dashboard 可以独立开发。

权威原则:

- durable store/event 是权威，live actor/session 只是运行时执行体。
- artifact ref 比文件路径更重要；路径是当前实现细节。
- dashboard 和 TUI 都读 read model/card，不直接拼内部实现状态。
- 迁移期 `feature_lanes.json` 是执行事实源；目标态它退化为 legacy projection。

测试边界:

- 每个事件有 pydantic/schema fixture。
- 每个 read envelope/card 有 golden fixture。
- CI 至少包含 schema compatibility、event replay idempotency、read envelope rendering
  三类 contract tests。

### 4. Dashboard 灰盒 Drill-Down

职责:

- 展开 compact card 的证据。
- 展示 lane graph、feature graph set、run health、review verdict、takeover context。
- 支撑排障和人工审查。

禁止:

- 作为用户默认主入口。
- 在常规用户流程中要求用户主动进入 dashboard 才能理解进度。
- 作为 workflow 的权威写入者。

测试边界:

- 使用 read model fixtures 和临时 artifact roots。
- 不依赖真实 TUI、Ray actor 或 LangGraph。
- Drill-down endpoint 必须能处理缺失 artifact，并返回明确 degraded/error shape。

## `feature_lanes.json` 迁移阶段

当前代码现实:

- `LaneStateMachine` 通过 `LaneProjectionSyncer` 读写 `feature_lanes.json`。
- `PlatformOrchestrator` 以它作为 lane dispatch 和状态更新入口。
- MCP lane mutation 工具和 dashboard lane endpoints 仍直接读写它。
- run health、read envelope、evidence aggregator 仍从它派生部分状态。

迁移分三阶段:

| 阶段 | 状态源 | 允许行为 | 禁止行为 |
|---|---|---|---|
| Stage 0: 当前 | `feature_lanes.json` 是执行事实源，graph-set 是上游 artifact | 通过 `LaneStateMachine`/`LaneProjectionSyncer` 写状态 | 直接手改 JSON；多 writer 抢写 |
| Stage 1: 双轨 | graph-set 是 planning 权威，`feature_lanes.json` 是 execution mirror | 写入必须带 `projection_source`、`graph_set_id`、`projection_revision` | TUI/dashboard 把 projection 当业务权威 |
| Stage 2: cutover | graph-native ready-set/status store 是执行权威，`feature_lanes.json` 仅兼容导出 | runner 从 graph-native source 读，旧 API 只读导出 | 新功能依赖 flat projection 字段 |

进入下一阶段的门槛:

- Stage 1 需要所有投影 lane 带 graph-set lineage 和状态 guard。
- Stage 2 需要 graph-native ready-set、status store、runner source flag 和 parity test。
- cutover 前协议中的“legacy projection”只能理解为目标态，不代表当前实现已完成。

## 最小事件契约

事件名采用 `<domain>.<verb>`。payload 必须包含可追踪 id、idempotency key 和
artifact refs。默认交付语义如下:

- 投递保证: at-least-once。
- 消费者要求: 幂等处理，按 idempotency key 去重。
- 顺序保证: 只保证同一 aggregate key 内单调版本；跨 aggregate 不保证全局顺序。
- 重试: transient failure 可重试；超过上限进入 dead-letter/read-model degraded。
- 乱序处理: 消费者必须比较 version / updated_at / projection_revision，拒绝旧事件覆盖新状态。

死信和升级路径:

- 监控者: xmuse coordinator 负责扫描 dead-letter queue / failed planning events /
  degraded read models。
- 记录: 每个 dead-letter 必须包含 source event ref、consumer、attempt count、last error、
  next suggested action。
- 用户可见性: 有 `conversation_id` 时，coordinator 发送 compact degraded card 到 GOD 群聊；
  无 conversation 时只写 dashboard/run-health degraded。
- 自动动作: 可判定为 transient infra failure 时按 backoff 重试；schema mismatch、
  impossible dependency、state guard mismatch 进入 blocked/degraded，等待 GOD/human
  显式重排或修复。
- dashboard: dead-letter 数量和最近错误必须出现在 run health / drill-down 中。

| 事件 | 生产者 | 消费者 | 交付保证 | 说明 |
|---|---|---|---|---|
| `conversation.created` | TUI/chat API | GOD runtime | at-least-once, idempotent by conversation_id | 新工作区/群聊创建 |
| `conversation.archived` | TUI/chat API | GOD runtime/read models | at-least-once, monotonic version | 工作区归档 |
| `message.posted` | TUI/GOD | GOD runtime/cards | at-least-once, ordered by conversation stream version | 群聊消息 |
| `blueprint.proposed` | GOD 群聊层 | TUI/dashboard | at-least-once, idempotent by blueprint_id+version | GOD 产出候选蓝图 |
| `blueprint.approved` | GOD 或人工开关 | Layer 2a | at-least-once, idempotent by resolution_id+version | 蓝图进入自动拆解 |
| `blueprint.rejected` | GOD 或人工开关 | TUI/read models | at-least-once, terminal per blueprint version | 讨论后放弃或要求重开 |
| `planning.started` | Layer 2a | feature planner/read models | at-least-once, idempotent by planning_run_id | 创建 planning run |
| `planning.failed` | Layer 2a | TUI/dashboard/GOD | at-least-once, terminal or retryable by reason | 自动拆解失败 |
| `feature_plan.ready` | Layer 2a | Layer 2b/TUI/dashboard | at-least-once, idempotent by feature_plan_id+version | feature plan 通过审查 |
| `graph_set.ready` | Layer 2b | Layer 2c/dashboard | at-least-once, idempotent by graph_set_id+version | graph-set 可执行 |
| `graph_set.failed` | Layer 2b | TUI/dashboard/GOD | at-least-once, retryable by graph_set_id+attempt | lane graph 生成失败 |
| `lane.ready` | Layer 2c projection | runner | at-least-once, idempotent by lane_id+projection_revision | lane 依赖满足 |
| `lane.updated` | runner/orchestrator | read models/dashboard/TUI | at-least-once, monotonic by lane version/revision | lane 状态变化 |
| `lane.blocked` | runner/orchestrator | TUI/dashboard/GOD | at-least-once, retryable or terminal by blocker type | 依赖、输入或环境阻塞 |
| `review.verdict` | review GOD | runner/read models | at-least-once, idempotent by verdict_id | 审查结果 |
| `takeover.requested` | review GOD/runner | takeover loop/GOD | at-least-once, idempotent by takeover_attempt_id | 需要接管 |
| `takeover.resolved` | takeover loop/GOD | runner/read models | at-least-once, monotonic by takeover_attempt_id | 接管完成 |
| `run.terminal` | runner | TUI/dashboard/gap review | at-least-once, terminal by graph/run id | graph/run 完成 |
| `blueprint.gap_found` | gap review | GOD 群聊/workflow | at-least-once, idempotent by gap_id | 实现后发现漏 feature/lane |

## 最小 Artifact 契约

| artifact | 当前实现位置 | 迁移期权威级别 |
|---|---|---|
| conversation/messages | `xmuse/chat.db` | 当前权威 |
| planning events | `xmuse/planning_events.sqlite3` | 当前权威 |
| planning runs | `xmuse/planning_runs.sqlite3` | 当前权威 |
| feature plans | `xmuse/feature_plans/` | planning 权威 |
| feature graph sets / lane graphs | `xmuse/lane_graphs/` | planning/graph 权威 |
| flat lane projection/status | `xmuse/feature_lanes.json` | Stage 0/1 执行事实源，Stage 2 兼容导出 |
| execution cards | read model/card store | 前端契约 |
| dashboard read models | `xmuse/read_models/` | 只读派生 |
| logs | `xmuse/logs/` | 证据，不是状态权威 |

Artifact 写入规则:

- 业务 artifact 必须有 stable id、version、created_at/updated_at、source refs。
- 从 artifact 派生 projection 时必须保留 lineage refs。
- read model 可重建；不可只把 read model 当唯一事实源。

## Ray 接入规则

Ray Core 只负责 actor 生命周期、隔离、重启和并发通信。

- Ray actor 可以承载 persistent GOD、worker coordinator、review peer。
- Ray actor 的内存不是权威状态。
- actor 崩溃后必须能从 durable store 恢复 identity、inbox、pending request。
- actor 间通信应落地为 message/event，不能只停留在内存调用。
- 先支持本机 Ray；分布式 Ray 是后续扩展。

测试边界:

- Ray backend 必须有 local fake actor backend。
- Actor crash/restart test 验证 durable store 恢复，不验证内存连续性。
- Ray 不可用时 native backend 必须能继续跑同一 contract suite。

## LangGraph 接入规则

LangGraph 只负责 workflow 编排。

- 节点输入输出应是 id、artifact ref、event ref，不应携带大块业务状态。
- 节点通过 xmuse store/service 写入权威 artifact。
- LangGraph 先以 shadow/replay 或可切换 backend 方式接入。
- native workflow 和 LangGraph workflow 必须产出等价 artifact trace。
- 禁止 LangGraph 绕过 xmuse state machine 直接改 lane status。

测试边界:

- LangGraph adapter 必须有 shadow/replay fixture。
- Native workflow 和 LangGraph workflow 的 artifact trace 要能比较。
- LangGraph import 不应成为 native path 启动必需条件。

## CLI Subagent / Skills 执行规则

执行开发阶段采用中心化 coordinator。coordinator 可以调用 Codex/OpenCode 等 CLI 的
subagent harness，但 subagent 只执行有边界的任务，不参与状态仲裁。

适合沉淀为 skills 的职责:

- lane 实现流程: 读上下文、写测试、实现、运行验证、提交结果摘要。
- blueprint acceptance 对齐检查。
- takeover context bundle 生成。
- merge failed / rework 排查。
- worker 输出格式和证据要求。
- review verdict 解读和修复建议。

仍必须由平台能力承载的职责:

- durable event / artifact store。
- feature plan、lane graph、graph-set 的创建和版本管理。
- lane ready-set 和状态机。
- provider/model selection。
- runner 写入仲裁和 state guard。
- dashboard/TUI read model materialization。

coordinator 对 subagent 的硬约束:

- 每个 subagent invocation 必须有 lane id、blueprint/feature refs、acceptance
  criteria、allowed files、verification command、输出 schema。
- subagent 输出必须回到 coordinator，由 coordinator 判断是否写状态。
- subagent 不直接调用 `update_lane_status`，不直接改 `feature_lanes.json`。
- subagent 失败时回传 blocker 分类、证据和可重试建议，不自行启动新链路。
- review subagent 可以给 verdict 建议，但最终 verdict artifact 和状态更新由
  coordinator/review gate 写入。

测试边界:

- skill contract tests 使用固定 prompt/context fixture 和 fake CLI harness。
- coordinator tests 验证: subagent 成功、subagent rework、subagent infra failure、
  subagent 输出 schema 不合法四类路径。
- 不要求每个 skill 都启动真实 CLI；真实 CLI 只进入 smoke 或 manual gate。

## TUI 与 Dashboard 的关系

- TUI 是主入口，展示群聊、compact cards、worklist 摘要。
- dashboard 是 card 的 drill-down 目标。
- dashboard API 可以服务 TUI，但不能迫使 TUI 理解 dashboard 内部页面结构。
- 新字段先进入 read envelope/card contract，再由 TUI 和 dashboard 各自渲染。

## 开发并行原则

1. Layer 1、2a、2b、2c、3、4 可以并行开发，只要不破坏本协议的事件和 artifact
   契约。
2. 新能力默认以可关闭 backend/adapter 引入，避免中途瘫痪 xmuse。
3. 优先扩展契约和 adapter，不把跨层逻辑堆进单个大文件。
4. 旧 browser frontend 作为历史/可选实现处理；当前用户前门优先 Textual TUI。
5. dashboard 改动应偏只读和 drill-down，不承担核心调度。
6. 若出现字段不一致，以本协议和当前代码中的 pydantic/store model 为准，更新
   文档后再推进实现。
7. 每层都必须能用 fake/stub 单独跑 contract tests；跨层 E2E 只验证关键路径。
8. 执行阶段优先复用 CLI subagent harness 和 skills，不为每个执行职责新增常驻
   对等 GOD。
